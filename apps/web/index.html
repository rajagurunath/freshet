The gap

Your best engineering decisions happen in AI sessions. Then the terminal closes.

Every day your team works through hard problems with Claude Code, Codex, and Kilo Code. The reasoning is sharp, specific, and earned.

But that knowledge never reaches the next engineer, the next agent, or the next time the same bug shows up. It's scattered across laptops, tools, and chat scrollback — and it's invisible.

Signal 01

Trapped context

The reasoning that solved a gnarly bug lives in one engineer's local transcript. Nobody else can find it or reuse it.

Signal 02

Fragmented tools

Claude Code here, Codex there, Kilo Code on another machine. No shared memory across the tools your team actually uses.

Signal 03

No institutional memory

New hires and new agents start cold. Decisions your team already made get re-litigated from scratch, every time.

Signal 04

Privacy dead-end

The obvious fix — ship every transcript to a vendor — is a non-starter. Your sessions hold secrets, keys, and proprietary code.

How it works

Capabilities

What a curated session corpus unlocks.

Every capability below is built on the same thing: the sessions your team chose to keep.

Hybrid retrieval

Exact identifiers and fuzzy meaning, together

Keyword (full-text) and vector search run in parallel and merge with reciprocal rank fusion — so ERR_5021 and "why does refresh 401" both find the right session.

Cross-session knowledge graph

Every session, and the whole picture

Each session gets its own entity graph; the same repo, service, library, or person mentioned across sessions becomes one shared node, linked by reversible same_as edges. Explore the whole graph or filter to the sessions that have one.

GraphRAG retrieval

Finds the session keyword search can't

A 6-hour session defeats grep. We walk the knowledge graph to surface sessions that share no words with your query but are linked by a shared entity. On a held-out benchmark, the graph arm lifts MRR +14% and takes bridge recall from 0 to 1.0 over plain vector+keyword search.

Built offline, in seconds

No cloud, no waiting

A native scanner reads only what the list needs, so hundreds of MB of history open instantly. Per-session graphs for your whole history build locally — hundreds of sessions in under two seconds — with a progress bar, entirely on your machine.

Branch from any turn

Fork a session at one message

Pick a turn, fork it into a fresh claude --resume lineage, and explore a different approach. The original transcript is never touched.

PR context links

Every PR carries its reasoning

Mint a token-gated page from a session and drop it on the PR. Reviewers see the why — the decisions, the linked issues — not just the diff.

Auto-sync & cheap batch

Opt in per tool, summarize for less

Closed sessions redact and push themselves on a quiet timer. Summaries run on the OpenAI Batch API (cheaper, off the hot path) or a local model if your machine can host one.

Rules you approve

Standardize what's already true

Recurring conventions — commit style, patterns, preferences — are mined from your sessions and proposed as rules. Nothing standardizes until you accept it; export the set as a CLAUDE.md.

The protocol · aicp/0.1

One open protocol for moving context between agents.

AICP — the AI Context Protocol — lets one agent pick up where another left off, pulling only as much as it needs. It's MCP-native, vendor-neutral, and built on progressive disclosure: five levels, smallest first.

Consumerany coding agent
  1. L0 session.list
  2. L1 session.summary
  3. L2 session.recent
  4. L3 session.grep
  5. L4 session.stream
Brokerdesktop app or hub
  1. L0
    Manifest. What sessions exist — titles, tools, token counts. No message bodies, almost free.
  2. L1
    Summary. The structured gist of a session, generated once and cached.
  3. L2
    Recent. The last N messages — the live working context, the part that's usually enough.
  4. L3
    Find-in-session. Keyword (and, later, semantic) matches inside one session, by message.
  5. L4
    Full stream. The complete transcript, streamed message-by-message and resumable from a cursor.

Consent & redaction, always. Every exchange honors the session's visibility scope, and redaction runs before any bytes leave. A read returns consent_required until you grant it.

One-shot handoff. session.handoff bundles summary + recent into a single packet — the common "hand this session to another agent" case, in one call.

Local broker first. AICP is specified as aicp/0.1; the first implementation is a local broker inside the desktop app — exposing every tool's sessions over MCP. The cloud broker is designed and comes next.

In practice

Workflows the hub makes possible.

Concrete things your team can do once sessions are captured and curated.

01

Hand a session to another agent

Roadmap · AICP v1

The goal AICP is built for: in Codex, "continue the auth refactor I did in Claude Code." Codex will ask your local broker and pull only what it needs.

session.search "auth refactor" → session.summary → session.recent

Lazy and context-budget-aware — reaching for grep or the full stream only when needed. The protocol is specced (aicp/0.1); the local broker that powers this is the next thing we're building. Today the desktop app already parses all three tools into one normalized view.

02

Put the reasoning on the pull request

Available now

Push the session, mint a share link, and drop it in the PR description — like a Linear ticket for the why.

Push to hub → Mint token-gated link → Reviewer opens the context page

The page shows the summary, the decision timeline, and linked issues — gated by a signed, expiring token. No login required to read it.

03

Spend your whole weekly subscription

Available now

Cloud plans reset on a schedule. Before the window closes, the harvester drains pending work so unused quota doesn't evaporate.

Detect reset is near → Summarize pending sessions → Extract knowledge graph

Time-based and best-effort — CLI providers don't expose a remaining-quota API, so the harvester works from the reset schedule you configure.
